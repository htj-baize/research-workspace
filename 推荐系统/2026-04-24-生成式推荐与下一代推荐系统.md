# 生成式推荐与下一代推荐系统

日期：2026-04-24

## 背景

这份笔记整理了一轮围绕生成式推荐、`Studio Quest`、Recall 架构，以及 Agent-native 推荐系统的讨论。

讨论的起点是一个很具体的问题：`Studio Quest` 到底应该被理解成什么。

结论不是：

- 重型任务树
- 世界评估器
- 付费弹窗系统

而是：

> 一个建立在世界状态之上的 continuation engine。

再往推荐系统语言里翻译，它可以被理解成：

> 一个 world-state-driven 的 generative continuation recommender。

## 1. `Studio Quest` 本质上在推荐什么

它推荐的不是传统推荐系统里的内容 item，也不是静态商品。

它推荐的是：

- 下一步继续创作什么
- 下一步最值得推进哪条线
- 下一步哪个 continuation offer 最应该被放到用户面前

也就是说，它推荐的是：

> 从当前世界状态里长出来的“下一步机会”。

因此，`Studio Quest` 不是在做：

- feed 分发
- item 排序
- 现成内容消费推荐

而是在做：

- continuation recommendation
- creative action recommendation
- world-state-driven offer generation

## 2. 为什么这是生成式推荐问题

它属于生成式推荐，原因有四个。

### 第一，推荐结果不是静态库存

这里最终展示给用户的 `ContinuationOffer`，不是内容池里已有的标准 item，而是根据：

- `world snapshot`
- 当前 atoms / works
- 最近 session
- 用户刚刚关注的对象

动态生成出来的。

### 第二，推荐依据是语义状态，不只是行为统计

系统要理解：

- 角色设定
- 关系张力
- 已生成作品
- 最近发生的 continuation
- 世界里哪些线索还没有兑现

这和传统的“相似用户也喜欢”不一样。

### 第三，推荐结果本身带生成表达

推荐不只是一个 ID。它还包含：

- `headline`
- `whyNow`
- `worldReason`

这些本身就是生成式内容。

### 第四，推荐会触发新的内容生成

用户点击 Quest 后，系统不是跳转到一个现成 item，而是：

- 进入 continuation run
- 生成新 work / variant / scene / atom variant
- 再把结果写回世界

这意味着推荐本身是创作闭环的一部分。

## 3. 它和传统推荐系统的差别

传统推荐更像：

```text
用户画像 + 行为历史 + item 池
-> recall item
-> rerank
-> show item
```

而 `Studio Quest` 更像：

```text
世界状态 + 当前 session + 当前 attention object
-> recall / generate quest possibilities
-> validity filter
-> translate to continuation offers
-> rank and price
-> user confirms
-> agent executes continuation
-> write back to world state
```

所以它不是“内容推荐”的简单变种，而是：

> 推荐系统、生成系统、状态系统、商业化系统的混合体。

## 4. 这里为什么需要 Recall

技术上，应该加入 Recall，但不是传统大规模推荐里的重型召回系统。

更准确地说，这里需要的是：

> 一个轻量、可解释、世界状态驱动的 candidate generation / recall 层。

因为如果没有 Recall，系统很容易退化成：

```text
每次直接让 LLM 即兴生成 3 个 Quest
```

这会带来几个问题：

- 候选不可解释
- 不知道哪些对象被考虑过
- 无法稳定做去重
- 很难做排序和 A/B 测试
- pricing 和 execution 难以拆层

所以结构上最好显式拆开：

```text
Quest Recall
-> Validity Filter
-> Offer Translation
-> Cost-aware Ranking
-> UI Display
-> Continuation Execution
-> World Writeback
```

## 5. 这里 Recall 的到底是什么

Recall 的不是：

- atom
- work
- 商品
- 静态模板
- 热门内容

Recall 的是：

> Quest candidate，也就是可能成立的下一步 continuation possibility。

在这套模型里，一个比较合适的 Recall 单位是：

```ts
type QuestPossibility = {
  id: string;
  dynamicType:
    | "tension"
    | "gap"
    | "promise"
    | "contradiction"
    | "escalation"
    | "resolution";
  sourceAtomIds: string[];
  sourceWorkIds: string[];
  worldReason: string;
  brief: string;
};
```

也就是说，Recall 层回答的是：

> 当前世界里有哪些“下一步”是成立的。

它还不是前台卡片，更不是最终展示内容。

## 6. MVP 阶段应该怎么做 Recall

MVP 不建议上来就做：

- embedding 大规模检索
- 协同过滤
- 跨用户相似人群推荐
- 全局热门 continuation

更适合的是 4 类轻量来源。

### 1. Recent Object Recall

围绕最近被创建、查看、编辑、continuation 的对象生成候选。

这是最重要的一层，因为用户的注意力刚刚停在这些对象上。

### 2. Unresolved Tension Recall

从当前世界里找：

- 秘密
- 矛盾
- 未兑现承诺
- 危险关系
- 缺失后果
- 可升级冲突

也就是从“还没完”的地方长 Quest。

### 3. Underdeveloped Object Recall

找那些已经存在、但还没有充分展开的对象：

- 有设定但没有 work 的角色
- 被多次引用但没有深化的对象
- 被暗示但没有场景承接的关系

### 4. Follow-up Pattern Recall

根据最近结果类型，提供一组下一步模式。

比如：

- 刚生成角色 -> first scene / flaw reveal / dangerous relationship
- 刚生成重逢场景 -> consequence / alternate POV / evidence
- 刚 continuation 一次 -> escalation / alliance / irreversible cost

但这些模式必须绑定当前对象和当前语境，不能变成通用模板按钮。

## 7. 什么才让 Quest 真正有趣

最关键的一点不是“文案更刺激”，而是：

> 它是否准确抓到了用户刚刚在乎的那个对象上的未完成欲望。

一个有用的工作公式是：

> 好 Quest = 具体对象 + 未完成张力 + 明确后果 + 可想象画面 + 低摩擦行动。

### 1. 贴住当前 attention object

如果用户刚生成的是“白槿”，那 Quest 应该围绕白槿，而不是突然推荐“扩展世界设定”。

### 2. 从未完成里长出来

最容易抓人的往往是这些未完成感：

- 她到底隐瞒了什么
- 这段关系会变成什么
- 这件事会造成什么后果
- 之前埋下的线索什么时候回来

### 3. 强 Quest 带后果，不只是动作

弱 Quest：

- 继续她的故事

强 Quest：

- 给她一次第一次露出破绽的场面

更强 Quest：

- 让这次重逢留下一个她再也解释不清的证据

### 4. Quest 要能被脑补成画面

没有画面的 Quest，容易像功能按钮。

有画面的 Quest，往往带：

- 场所
- 动作
- 人物关系
- 具体后果

### 5. 不要只推最强刺激，要推“现在最该发生”

真正抓人的不是强度最大，而是时机最对。

所以排序里很重要的一项其实是：

> Narrative Timing

## 8. 核心技术难点到底是什么

核心难点不是“生成 3 张卡片”，而是能不能把下面这条链路跑顺：

```text
世界状态
-> 长出 continuation possibility
-> 翻译成可卖的 offer
-> 用户点击
-> 生成真正有价值的新内容
-> 写回世界
-> 再长出下一轮 offer
```

这里面最关键的 3 件事是：

### 1. Quest 必须绑定具体对象

如果一个 Quest 说不清自己是从哪个 atom / work 长出来的，它大概率就是假的。

### 2. Offer 必须可执行

前台 headline 好看不够，后台必须能把它转成：

- continuation intent
- 使用哪些 source
- 输出什么类型
- 写回哪里

### 3. 结果必须能回写并支持下一轮

如果 continuation 只是生成一段漂亮文本，但不能稳定写回 `works` 或 `atoms`，整个系统就会停在第一轮。

## 9. 未来推荐系统会退回 `profile + recall + rerank` 吗

不会简单退回去。

更可能的趋势是：

> 系统重新分层，但不是回到老式推荐架构，而是演化成 agent-orchestrated recommendation stack。

纯端到端推荐的问题在于：

- 可控性不够
- 工业约束难内化
- 成本和延迟难稳定
- 难 debug
- 难做多目标权衡

但传统 `profile + recall + rerank` 也不会原样回来。

它们会被重新定义：

### `profile` 变成动态用户状态

不再只是静态画像，而更像：

- long-term preference
- current intent
- memory
- context state

### `recall` 变成 candidate construction

不只是在静态 item 池里找东西，还包括：

- 检索
- 组装
- 生成
- 动态构造下一步候选

### `rerank` 变成 policy layer

它不只是打分，而是在做：

- 价值判断
- 风险控制
- 成本控制
- 多目标平衡

所以更像：

```text
user state / memory
+ retrieval / candidate construction
+ policy / ranking
+ agent planning
+ action execution
+ writeback / memory update
```

## 10. 如果一切都 agent-native 化，人的价值在哪

人的价值不会消失，而是会往上移。

未来稀缺的不是“能不能生成东西”，而是：

> 能不能持续地产出对的东西，并对结果负责。

人更核心的价值会在：

- 定义问题
- 定义目标
- 设定约束
- 设计系统演化方式
- 做 eval 和校准
- 对后果负责

Agent 更擅长：

- 展开候选
- 局部执行
- 处理重复性工作
- 在清晰目标下快速优化

所以边界不在“谁写代码”，而在：

> 谁定义系统应该优化什么，谁承担系统出错后的真实责任。

## 11. 在下一代推荐系统里，人和 Agent 分别是什么角色

一个比较清楚的划分是：

### Agent 是动态决策与执行层

Agent 更像：

- intent interpreter
- candidate constructor
- interaction manager
- action executor

它负责：

- 理解用户当前要做什么
- 补充澄清意图
- 构造候选内容 / 候选动作 / 候选路径
- 执行推荐后的 workflow
- 更新 session 和 memory

### 人是价值、政策与责任层

人更像：

- objective designer
- policy owner
- evaluator
- system owner

人负责：

- 定义系统到底优化什么
- 决定哪些边界不能碰
- 设计长期指标
- 处理目标冲突
- 为系统行为承担责任

在推荐系统里，这一点尤其重要。因为 agent 很容易朝着：

- 短期点击
- 局部最优
- 高刺激
- 高利润

这些方向滑过去，而人需要保证系统不会因此伤害：

- 用户体验
- 内容生态
- 商业长期性
- 安全和合规

## 12. 对生成式推荐的一个简洁判断

如果用一句话总结这轮讨论，我会写成：

> 下一代推荐系统不会消失在端到端模型里，也不会简单退回传统三段式，而会演化成由 Agent 驱动、由状态和记忆支撑、由策略层约束、并由人来定义价值与责任的推荐系统。

对 `Studio Quest` 这样的产品来说，这个判断会更具体：

> 它不是传统内容推荐，而是基于当前世界状态，为用户生成并筛选“下一步最值得继续”的创作机会。

这类系统最关键的不是多会说，而是：

- 生成的候选是否真从当前状态里长出来
- 排到前台的 offer 是否真能抓住用户
- 点击后是否真能产出可回写的新内容
- 回写后是否还能继续生长

也就是：

> generative recommendation 的核心，不只是生成推荐，而是生成一个可持续演化的行动闭环。
